# BPERP Project Context & AI Instructions

> Internal manufacturing ERP for Bray Precision LLC
> Windows & Linux deployment | Express.js + PostgreSQL + Electron
> **Note**: Internal tool only. All workstations share a network PostgreSQL database. No licensing, multi-tenant, or commercial features.

---

## Core Principles

### Code Quality Philosophy
- **Write clean, maintainable, and scalable code** - This system will grow with the business
- **Prioritize readability over "clever" one-liners** - Code is read more than written
- **Follow DRY but avoid premature abstraction** - Don't over-engineer until patterns emerge
- **Think manufacturing workflow** - Accuracy and reliability are critical; a wrong quantity or price can cost real money
- **Performance matters** - Efficient queries, minimal re-renders, fast response times

### Technology Boundaries
- Frontend stays **vanilla JavaScript** (no React/Vue/Angular)
- Backend uses JavaScript routes (TypeScript migration is not a priority — JS routes work fine)
- No frontend build step - works directly in browser
- **Network database** - all workstations connect to a shared PostgreSQL server on the LAN
- **Windows & Linux only** - no macOS support
- Internal use only - no licensing, multi-tenant, or commercial distribution features
- **Auto-refresh** - workcenter displays should stay current without manual refresh
- **User profiles from DB** - appearance, permissions, and settings load from the database on login; no per-device manual setup

### Coding Style
- Use **functional programming patterns** where appropriate
- Prefer **descriptive variable names**: `isInventoryLoading` not `loading`
- Constants in `UPPER_SNAKE_CASE`
- Database columns: `snake_case` (e.g., `customer_id`)
- API responses: `camelCase` (e.g., `customerId`)
- Transform between formats in route handlers

Electron app which starts a local Express backend that connects to the network database.

## Backend Patterns

### Route Factory Pattern
All routes export a factory function receiving the database pool:

```javascript
// backend/routes/example.js
module.exports = function(pool) {
    const router = require('express').Router();
    const { requireAuth } = require('../middleware/auth')(pool);

    router.get('/', requireAuth, async (req, res) => {
        const result = await pool.query('SELECT * FROM table');
        res.json({ success: true, data: result.rows });
    });

    return router;
};
```

### API Response Format
```javascript
// Success
{ success: true, data: T }
{ success: true, data: T[], pagination: { page, limit, total, totalPages } }

// Error
{ success: false, error: "Error message" }
```

### Middleware Stack
```javascript
const { requireAuth, requireAdmin, requireRole } = require('../middleware/auth')(pool);
const { validate, schemas } = require('../middleware/validation');
const { apiLimiter } = require('../middleware/rateLimit');

router.post('/',
    requireAuth,                    // Token validation
    validate(schemas.customer),     // Zod validation
    apiLimiter,                     // Rate limiting
    handler
);
```

### Authentication
- Token in `Authorization: Bearer <token>` header
- Stored in `user_sessions` table with expiration
- Roles: `Administrator`, `Machinist`, `Operator`
- Passwords: bcrypt (12 rounds)

## Common Tasks

### Adding a New API Route
1. Create `backend/routes/newroute.js` using factory pattern
2. Add route mounting in `backend/server.js`:
   ```javascript
   app.use('/api/newroute', require('./routes/newroute')(pool));
   ```
3. Add Zod schema in `backend/middleware/validation.js`
4. Add rate limiting if sensitive

### Adding a Frontend Module
1. Create `frontend/js/modules/newmodule.js`
2. Export `init()` and view functions
3. Import in `frontend/js/app.js`
4. Add to `window.BPERP` namespace
5. Add navigation in `frontend/index.html` sidebar

### Database Migration
1. Create migration in `backend/migrations/scripts/YYYYMMDD_HHMMSS_description.js`
2. Export `up(pool)` and `down(pool)` functions
3. Run: `cd backend && npm run migrate`

### Building Installers
```bash
# Ensure backend dependencies installed
npm run backend:install

# Build for Windows
npx electron-builder --win --publish never

# Build for Linux
npx electron-builder --linux deb AppImage --publish never
```

## Manufacturing-Specific Patterns

This is a **manufacturing ERP** - domain knowledge matters:

### Data Accuracy
- **Quantity validations** must consider decimal places (you can have 2.5 feet of material)
- **Price calculations** use proper decimal handling - never floating point for money
- **Part numbers and work order numbers** are critical identifiers - treat like primary keys
- **Inventory counts** must be exact - audit trails for all changes

### Status Workflows
Statuses reflect real manufacturing processes:
```
Quote: Draft -> Sent -> Won/Lost
Work Order: Pending -> In Progress -> QC -> Complete -> Shipped
Task: Pending -> In Progress -> Blocked -> Complete
Maintenance: Scheduled -> In Progress -> Complete -> Deferred
```

### Audit Requirements
- Log all inventory quantity changes with user, timestamp, reason
- Track quote revisions and who made changes
- Maintain work order history for quality traceability

## Error Handling & Safety

### Backend
- **Wrap all async operations** in try/catch blocks
- **Never expose internal errors** to the client - log details, return generic message
- **Log errors with context** but not sensitive data (no passwords, tokens)
- **Use transactions** for multi-table operations that must succeed together

```javascript
try {
    await pool.query('BEGIN');
    // ... multiple operations
    await pool.query('COMMIT');
} catch (error) {
    await pool.query('ROLLBACK');
    console.error('Operation failed:', error.message);
    res.status(500).json({ success: false, error: 'Operation failed' });
}
```

### Frontend
- **Handle Loading states** - show spinners, disable buttons during async ops
- **Handle Error states** - display user-friendly messages, offer retry
- **Validate inputs client-side** before sending to server (but always validate server-side too)

## Security Considerations

### Authentication & Authorization
- **RBAC** (Role-Based Access Control) with three roles
- **Validate inputs on frontend AND backend** - never trust client data
- **Rate limiting** on sensitive endpoints (login, user creation, exports, imports)
- **bcrypt** for password hashing (12 salt rounds)
- **Token expiration** - sessions expire, require re-login

### Data Protection
- **No sensitive data in localStorage** - only in server sessions
- **Parameterized queries only** - prevent SQL injection
- **CORS configured** for specific origins
- **Sanitize HTML** when rendering user-provided content
