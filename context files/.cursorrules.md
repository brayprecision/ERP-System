# BPERP Project Standards & AI Instructions

## 1. Project Context
- This is BPERP (Bray Precision ERP) - an on-premise manufacturing ERP system
- **Backend**: Express.js 4.18 + PostgreSQL 14+ + TypeScript (in progress)
- **Frontend**: Vanilla ES6 JavaScript + Tailwind CSS (via CDN)
- Key features: Inventory tracking, job management, quoting system, task management, work orders, maintenance
- Target market: Small machine shops (1-20 employees)
- Deployment: On-premise (customer installs on their own hardware)

## 2. General Principles
- Write clean, maintainable, and scalable code
- Prioritize readability over "clever" one-liners
- Follow "Don't Repeat Yourself" (DRY) but avoid premature abstraction
- Think manufacturing workflow: accuracy and reliability are critical
- Frontend stays vanilla JavaScript (no React/Vue/Angular)
- Backend is migrating to TypeScript incrementally

## 3. TypeScript Standards (Backend)
- TypeScript files go in `backend/src/`, compiled to `backend/dist/`
- Use type inference where obvious, explicit types for function returns
- Prefer interfaces over type aliases for object shapes
- All entity types defined in `backend/src/types/index.ts`
- Database row types (snake_case) in `backend/src/types/database.ts`
- Zod schemas provide runtime validation + type inference

Example:
```typescript
// Route handler typing
router.get('/:id', async (
    req: Request<{ id: string }>,
    res: Response<ApiResponse<Customer>>
) => { ... });
```

## 4. Coding Style
- Use functional programming patterns where appropriate
- Prefer descriptive variable names (e.g., `isInventoryLoading` not `loading`)
- Constants in UPPER_SNAKE_CASE
- Database columns: snake_case (e.g., `customer_id`)
- API responses: camelCase (e.g., `customerId`)
- Transform between formats in route handlers

## 5. Backend Patterns
- Routes in `backend/routes/` (JavaScript) or `backend/src/routes/` (TypeScript)
- Middleware in `backend/middleware/` or `backend/src/middleware/`
- Use factory pattern for routes: `module.exports = function(pool) { ... }`
- All routes use PostgreSQL pool passed via factory
- Rate limiting on sensitive endpoints (login, create user, export, import)
- Input validation with Zod middleware

## 6. Frontend Patterns
- Modules in `frontend/js/modules/` (ES6 modules)
- Main app entry in `frontend/js/app.js`
- State stored in localStorage via `storage.js` module
- Use event delegation where possible
- Tailwind CSS for styling (via CDN, no build step)
- Font Awesome for icons

## 7. API Conventions
- RESTful endpoints: `/api/[resource]/[id]`
- Consistent response format: `{ success: boolean, data?: T, error?: string }`
- Use proper HTTP status codes (200, 201, 400, 401, 403, 404, 500)
- Validate request bodies with Zod schemas
- Include pagination for list endpoints: `{ data, pagination: { page, limit, total, totalPages } }`
- Authentication via Bearer token in Authorization header

## 8. Database & Data Handling
- Always use parameterized queries (prevent SQL injection)
- Implement soft deletes for critical data (`deleted_at` field)
- Use transactions for multi-table operations
- Timestamps: `created_at`, `updated_at` on all tables
- JSONB for flexible fields (appearance_settings, tab_permissions)
- Use migrations for schema changes: `npm run migrate`

## 9. Authentication & Authorization
- Token-based authentication (stored in `user_sessions` table)
- Password hashing with bcrypt (12 salt rounds)
- Rate limiting on login (5 attempts per 15 minutes)
- Role-based access: Administrator, Machinist, Operator
- Tab-level permissions stored per user
- Middleware: `requireAuth`, `requireAdmin`, `requireSelfOrAdmin`

## 10. Error Handling & Safety
- Wrap all async operations in try/catch blocks
- Use Zod for schema validation at API boundaries
- Handle Loading and Error states in frontend
- Never expose internal error details to the client
- Log errors with context (but not sensitive data)

## 11. Manufacturing-Specific Patterns
- Quantity validations consider decimal places
- Price calculations use proper decimal handling
- Status workflows reflect real manufacturing processes
- Part numbers and work order numbers are critical identifiers
- Audit trails for inventory changes and user actions

## 12. Testing
- Jest for unit and integration tests
- Supertest for API endpoint testing
- Test files in `backend/tests/`
- Run tests: `npm test`
- Unit tests don't require database
- Integration tests require PostgreSQL test database

## 13. File Structure

```

├── routes/            # JS routes (runtime)
├── src/               # TypeScript source
│   ├── types/         # Type definitions
│   ├── middleware/    # TS middleware
│   └── routes/        # TS routes
├── dist/              # Compiled output
├── migrations/        # Database migrations
├── tests/             # Jest tests

├── js/
│   ├── app.js         # Main app entry
│   └── modules/       # Feature modules

├── assets/
```

## 14. Security Considerations
- RBAC (Role-Based Access Control)
- Validate all inputs (frontend AND backend)
- Rate limiting on sensitive endpoints
- CORS configured for specific origins
- No sensitive data in localStorage (only in server session)
- bcrypt for password hashing

## 15. AI Assistant Instructions
- Consider the Express.js + vanilla JS stack (NOT Next.js/React)
- TypeScript is backend-only and migration is in progress
- Include proper error handling for all async operations
- For database changes, provide migration scripts
- Consider the on-premise deployment model
- Frontend code must work without a build step
- Use the existing middleware and type definitions

## 16. Code Review Checklist
Before finalizing any code:
- [ ] Error handling is comprehensive
- [ ] Input validation present
- [ ] SQL queries are parameterized
- [ ] Follows naming conventions (camelCase API, snake_case DB)
- [ ] TypeScript types properly defined (no 'any')
- [ ] Rate limiting on sensitive endpoints
- [ ] Authorization checks where needed
