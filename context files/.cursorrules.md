# BPERP Project Standards & AI Instructions

## 1. Project Context
- This is BPERP (Bray Precision ERP) - an internal manufacturing ERP system
- Primary tech stack: Next.js 14+, TypeScript, Tailwind CSS, PostgreSQL
- Key features: Inventory tracking, job management, quoting system, task management
- Users: Internal staff at Bray Precision LLC, Lakeport CA

## 2. General Principles
- Write clean, maintainable, and scalable TypeScript code
- Prioritize readability over "clever" one-liners
- Follow "Don't Repeat Yourself" (DRY) but avoid premature abstraction
- All code must be type-safe. Avoid 'any' at all costs
- Think manufacturing workflow: accuracy and reliability are critical

## 3. TypeScript Standards
- Use type inference where obvious, explicit types for function returns
- Prefer interfaces over type aliases for object shapes
- Use const assertions for literal types
- Use enums sparingly, prefer const objects with 'as const'
- Example: `const STATUS = ['pending', 'active', 'completed'] as const`
- Create a types file for each feature module

## 4. Coding Style
- Use functional programming patterns; avoid 'class' keywords unless necessary
- Use arrow functions for components and hooks
- Prefer descriptive variable names (e.g., `isInventoryLoading` not `loading`)
- Use kebab-case for filenames (e.g., `inventory-alert-card.tsx`)
- Constants in UPPER_SNAKE_CASE
- Group imports: React → Next → Third-party → Local → Types

## 5. React & Next.js Patterns
- Use Next.js App Router conventions
- Server Components by default, Client Components only when needed ('use client')
- Use Tailwind CSS for styling; follow mobile-first approach
- Separate business logic from UI (use custom hooks for complex logic)
- Use Lucide-react for icons and shadcn/ui for components
- Implement loading.tsx and error.tsx for route segments
- Use Next.js Image component for all images
- Implement proper metadata for SEO (even for internal apps)

## 6. State Management
- Use React Query (TanStack Query) for server state
- Use Zustand for complex client state, Context API for simple cases
- Keep state as close to where it's used as possible
- Avoid prop drilling - use composition or state management
- Cache API responses appropriately (consider stale-while-revalidate)

## 7. API Conventions
- RESTful endpoints: `/api/[resource]/[id]`
- Consistent response format: `{ data: T, error?: string, message?: string }`
- Use proper HTTP status codes (200, 201, 400, 401, 404, 500)
- Always validate request bodies with Zod schemas
- Include pagination for list endpoints: `{ data: T[], total: number, page: number, limit: number }`
- Use middleware for authentication and common headers

## 8. Database & Data Handling
- Always use parameterized queries (prevent SQL injection)
- Implement soft deletes for critical data (add `deleted_at` field)
- Use transactions for multi-table operations
- Index frequently queried columns
- Normalize data but consider read performance
- Use timestamps: `created_at`, `updated_at` for all tables

## 9. Error Handling & Safety
- Wrap all async operations in try/catch blocks
- Use Zod for schema validation at API boundaries
- Implement proper error boundaries for component trees
- Handle Loading and Error states for all data fetching
- Use optional chaining (?.) and nullish coalescing (??)
- Log errors with context (user, action, timestamp)
- Never expose internal error details to the client

## 10. Performance Guidelines
- Use `useMemo` and `useCallback` only when there's a proven bottleneck
- Implement dynamic imports for heavy components
- Lazy load routes and heavy dependencies
- Optimize images (WebP format, appropriate sizes)
- Implement virtual scrolling for long lists
- Use debouncing for search inputs and auto-save features
- Monitor and optimize database queries (use EXPLAIN)

## 11. Manufacturing-Specific Patterns
- Quantity validations must consider decimal places (some materials use fractional quantities)
- Implement audit trails for inventory changes
- Price calculations should use decimal types, not floats
- Date handling must consider shop floor schedules
- Status workflows should reflect real manufacturing processes
- Part numbers and job numbers are critical identifiers - validate format
## 11.1 Inventory Thresholds
- Critical Low: < 5 units (red)
- Reorder Soon: < 10 units (orange)  
- Low Stock: configurable per item (orange)
- Adequate Stock: above reorder point (green)

## 12. Testing Approach
- Write tests for critical business logic (calculations, inventory updates)
- Use React Testing Library for component tests
- Test user workflows, not implementation details
- Integration tests > unit tests for API endpoints
- Test data validation and error cases
- Mock external services appropriately

## 14. Component Patterns
- Each component in its own file
- Co-locate component-specific types and utilities
- Use compound components for complex UI (Table.Header, Table.Body)
- Implement proper ARIA labels for accessibility
- Export types alongside components when needed

## 15. Security Considerations
- Implement RBAC (Role-Based Access Control)
- Validate all user inputs (frontend AND backend)
- Use HTTPS everywhere, even internally
- Implement CSRF protection
- Regular dependency updates
- Audit log sensitive operations

## 16. AI Assistant Instructions
- When providing code, include comments for potential edge cases using FIXME: or NOTE:
- For complex features, provide a 2-3 sentence implementation plan first
- Always specify where code should be placed in the file structure
- Include TypeScript types for all function parameters and returns
- When suggesting database changes, provide migration scripts
- Consider manufacturing domain knowledge in solutions
- Run a mental "lint check" before providing final code
- For debugging: ask for error messages, expected vs actual behavior, and relevant code
- Use TODO: for items that need future attention
- Provide error handling for all async operations
- Consider mobile responsiveness in all UI suggestions

- Never commit secrets to version control
- Use strong typing for env vars validation
